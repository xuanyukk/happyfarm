## 1. 概述

### 1.1 设计目标

本方案旨在为开心农场后台管理系统拓展五个核心功能，提升系统的管理能力、运营效率和实时监控水平：

- **精细化权限管理**：实现RBAC权限模型，支持功能级、操作级、数据级权限控制
- **完善公告系统**：支持富文本编辑、分类管理、定时发布、阅读跟踪
- **灵活参数配置**：游戏参数可视化配置，支持版本管理和回滚
- **高效批量操作**：统一批量操作接口，支持异步处理和进度跟踪
- **实时预警推送**：WebSocket长连接推送，支持自定义规则和定向推送

### 1.2 设计原则

1. **架构一致性**：严格遵循现有项目架构和代码规范
2. **功能可扩展性**：模块化设计，便于未来功能扩展
3. **用户体验优化**：直观的操作界面，清晰的业务流程
4. **安全与审计**：所有操作记录审计日志，关键操作走审批流程
5. **性能优先**：大数据量操作支持异步处理，不阻塞主流程

```
玩家管理
├── 玩家查看 (player:view)
├── 玩家禁用 (player:disable)
└── 玩家信息修改 (player:modify)

├── 公告管理 (game:announcement)
```

### 2.3 数据库设计

#### 2.3.1 核心数据表

```sql
-- 角色表
CREATE TABLE admin_role (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL COMMENT '角色名称',
    description TEXT COMMENT '角色描述',
    is_system BOOLEAN DEFAULT FALSE COMMENT '是否系统内置角色',

    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

-- 权限表
CREATE TABLE admin_permission (
    id SERIAL PRIMARY KEY,
    code VARCHAR(100) UNIQUE NOT NULL COMMENT '权限代码（如：sys:user:view）',
    name VARCHAR(50) NOT NULL COMMENT '权限名称',
    description TEXT COMMENT '权限描述',
    module VARCHAR(50) NOT NULL COMMENT '所属模块',
    level VARCHAR(20) NOT NULL DEFAULT 'function' CHECK (level IN ('function', 'operation', 'data')) COMMENT '权限级别',
    is_system BOOLEAN DEFAULT FALSE COMMENT '是否系统内置',
    parent_id INTEGER REFERENCES admin_permission(id) COMMENT '父权限ID',
    sort_order INTEGER DEFAULT 0,
    is_active BOOLEAN DEFAULT TRUE COMMENT '是否启用',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 角色-权限关联表
CREATE TABLE role_permission (
    id SERIAL PRIMARY KEY,
    role_id INTEGER REFERENCES admin_role(id) ON DELETE CASCADE,
    permission_id INTEGER REFERENCES admin_permission(id) ON DELETE CASCADE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(role_id, permission_id)
);

-- 用户-角色关联表
CREATE TABLE user_role (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES admins(id) ON DELETE CASCADE,
    role_id INTEGER REFERENCES admin_role(id) ON DELETE CASCADE,
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, role_id)
);

-- 数据权限表
CREATE TABLE data_permission (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES admins(id),
    role_id INTEGER REFERENCES admin_role(id),
    data_scope VARCHAR(20) NOT NULL DEFAULT 'OWN' CHECK (data_scope IN ('ALL', 'DEPARTMENT', 'OWN', 'CUSTOM')),
    custom_rule JSONB COMMENT '自定义数据规则',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 权限变更审计表
CREATE TABLE permission_audit_log (
    id SERIAL PRIMARY KEY,
    operator_id INTEGER REFERENCES admins(id),
    operation_type VARCHAR(30) NOT NULL CHECK (operation_type IN ('ASSIGN', 'REVOKE', 'CREATE', 'DELETE')),
    target_type VARCHAR(30) NOT NULL COMMENT '操作目标类型',
    target_id INTEGER NOT NULL COMMENT '操作目标ID',
    old_value JSONB COMMENT '变更前',
    new_value JSONB COMMENT '变更后',
    reason TEXT COMMENT '变更原因',
    ip_address VARCHAR(50),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2.5 前端实现要点

#### 2.5.1 权限控制组件

```javascript
// directives/permission.js
import { useAdminStore } from '@/stores/admin';

export default {
    mounted(el, binding) {
        const { value } = binding;
        const adminStore = useAdminStore();
        
        if (value && value instanceof Array && value.length > 0) {
            const hasPermission = value.some(perm => 
                adminStore.hasPermission(perm)
            );
            if (!hasPermission) {
                el.parentNode && el.parentNode.removeChild(el);
            }
        }
    }
}
```

#### 2.5.2 路由守卫

```javascript
// router/index.js
router.beforeEach(async (to, from, next) => {
    if (to.meta.requiresPermission) {
        const adminStore = useAdminStore();
        const hasPermission = await adminStore.hasPermission(to.meta.requiresPermission);
        if (!hasPermission) {
            next('/admin/no-permission');
            return;
        }
    }
    next();
});
```

| **用户体验** | 预热机制确保首屏加载速度 |
| **可观测性** | 缓存统计帮助定位性能问题 |

## 15. 技术架构统一要求

### 15.1 代码规范

1. **后端规范**
   - 严格遵循现有Express架构
   - 使用Winston统一日志记录
   - API响应格式一致
   - 中间件统一使用

2. **前端规范**
   - Vue3组合式API
   - Pinia状态管理
   - 统一错误处理
   - 组件按规则命名

3. **数据库规范**
   - 使用pg-pool连接池
   - SQL注入防护
   - 索引优化
   - 分区设计（大数据表）

### 15.2 安全要求

1. **认证与授权**
   - Token认证保持一致
   - 新增功能必须走权限控制
   - 关键操作审计日志

2. **数据安全**
   - 用户输入严格验证
   - XSS防护
   - SQL注入防护
   - CSRF保护

### 15.3 性能要求

1. **后端优化**
   - 数据库查询优化
   - Redis缓存热点数据
   - 异步处理耗时操作

2. **前端优化**
   - 组件懒加载
   - 虚拟滚动（大数据列表）
   - WebWorker处理计算密集任务
   - WebSocket推送替代轮询
