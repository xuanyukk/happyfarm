## 一、P0 紧急漏洞（已全部修复）

### 1.1 限流器内存/连接泄漏

| 字段 | 详情 |
|------|------|
| **漏洞编号** | SEC-001 |
| **风险等级** | 🔴 严重 |
| **文件位置** | `backend/src/middleware/rateLimiter.js:134-145` |
| **漏洞类型** | 资源泄漏（内存/Redis连接） |
| **CVSS 评分** | 7.5 (High) |

**漏洞描述**：  
`softRateLimiterMiddleware` 每次请求调用时都通过 `new RateLimiterRedis()` 创建新的限流器实例。每个实例都会创建到 Redis 的新连接，高并发场景下导致 Redis 连接数爆炸（每个请求占用 1 个连接），同时内存中积累大量未释放的限流器对象。

**影响范围**：
- 所有经过 `softRateLimiter` 中间件的公共接口（登录、注册等）
- Redis 服务端连接数耗尽，导致其他服务无法连接 Redis
- Node.js 进程内存持续增长，可能触发 OOM 被系统杀死

**修复方案**：  
将限流器实例从「每次请求创建」改为「模块级单例」，在 `initRateLimiter()` 和 `initMemoryLimiter()` 中一次性创建，中间件中复用。

**修复状态**：✅ 已修复

### 1.2 数据库密码明文日志泄露

| 字段 | 详情 |
|------|------|
| **漏洞编号** | SEC-002 |
| **风险等级** | 🔴 严重 |
| **文件位置** | `backend/src/config/db.js:25-28` |
| **漏洞类型** | 敏感信息泄露 |
| **CVSS 评分** | 8.6 (High) |

**漏洞描述**：  
`logger.debug` 直接输出 `process.env.DATABASE_URL` 的完整值，其中包含数据库主机地址、端口、用户名和密码明文。若日志系统被攻击者访问（如通过 Grafana Loki 查询、日志文件泄露等），数据库凭证将直接暴露。

**影响范围**：
- 数据库全部数据面临被窃取风险
- 攻击者获得数据库凭证后可执行任意 SQL 操作
- 若数据库密码在其他系统复用，攻击面扩大

**修复方案**：  
移除日志中的 `DATABASE_URL` 完整输出，改为仅输出 `typeof` 和 `defined` 状态。

**修复状态**：✅ 已修复

### 1.3 道具函数字段名错误导致运行时崩溃

| 字段 | 详情 |
|------|------|
| **漏洞编号** | SEC-003 |
| **风险等级** | 🔴 严重 |
| **文件位置** | `backend/src/services/itemService.js:944/981/1180` |
| **漏洞类型** | 代码缺陷（字段名错误） |
| **CVSS 评分** | 6.5 (Medium) |

**漏洞描述**：  
三个道具使用函数中 SQL WHERE 条件使用了 `user_id` 字段，但 `player_base` 表的主键是 `player_id`，`user_id` 字段不存在：
- `useWateringBoost` (L944)：`WHERE user_id = $3`
- `useFertilizeBoost` (L981)：`WHERE user_id = $1`（SELECT 查询）
- `useSkin` (L1180)：`WHERE user_id = $2`

**影响范围**：
- 玩家使用浇水加速、施肥加速、皮肤道具时直接报错
- 道具被消耗但效果未生效，导致玩家道具损失
- 属于二期新增道具的功能缺陷

**修复方案**：  
将 3 处 `user_id` 统一改为 `player_id`，与 `player_base` 表主键字段名一致。

**修复状态**：✅ 已修复

### 1.4 unlockLand 空结果集崩溃

| 字段 | 详情 |
|------|------|
| **漏洞编号** | SEC-004 |
| **风险等级** | 🔴 严重 |
| **文件位置** | `backend/src/services/farmService.js:79` |
| **漏洞类型** | 空指针崩溃 |
| **CVSS 评分** | 7.5 (High) |

**漏洞描述**：  
`unlockLand` 函数中直接访问 `statusResult.rows[0].is_unlocked` 和 `currencyResult.rows[0].currency_num`，未检查 `rows.length === 0`。当查询结果为空时抛出 `TypeError: Cannot read properties of undefined`，导致 Node.js 进程崩溃（若未捕获）。

**影响范围**：
- 玩家解锁地块操作可能触发服务端崩溃
- 影响所有在线玩家的游戏体验

**修复方案**：  
在访问 `rows[0]` 属性前添加 `rows.length === 0` 检查，抛出明确的业务错误。

**修复状态**：✅ 已修复

### 1.6 WebSocket 重连后丢失事件订阅

| 字段 | 详情 |
|------|------|
| **漏洞编号** | SEC-006 |
| **风险等级** | 🔴 严重 |
| **文件位置** | `frontend/src/services/websocketService.js:77-82` |
| **漏洞类型** | 功能缺陷（事件丢失） |
| **CVSS 评分** | 5.3 (Medium) |

**漏洞描述**：  
WebSocket 重连事件中只重新发送 `authenticate` 认证消息，但 Socket.IO 新连接会丢失之前通过 `socket.on()` 注册的所有事件监听器。导致重连后 UI 无法收到任何实时更新（作物生长、资源变化等）。

**影响范围**：
- 网络波动后用户看到的数据不再实时更新
- 用户需要手动刷新页面才能恢复

**修复方案**：  
添加 `restoreHandlers()` 方法，在 `reconnect` 事件中遍历 `messageHandlers` Map，重新绑定所有已注册的事件监听器。

**修复状态**：✅ 已修复

### 1.7 离线缓存同步为模拟代码

| 字段 | 详情 |
|------|------|
| **漏洞编号** | SEC-007 |
| **风险等级** | 🔴 严重 |
| **文件位置** | `frontend/src/services/offlineCache.js:275-312` |
| **漏洞类型** | 功能缺陷（未实现） |
| **CVSS 评分** | 6.5 (Medium) |

**漏洞描述**：

实现 `executeOperation` 方法，建立操作类型到 API 调用的映射表（plant/harvest/water/fertilize/unlockLand/upgradeLand/buyItem/sellItem/useItem），在同步时调用真实 API。

**修复状态**：✅ 已修复

### 1.8 数据库查询缓存失效 BUG

| 字段 | 详情 |
|------|------|
| **漏洞编号** | SEC-008 |
| **风险等级** | 🔴 严重 |
| **文件位置** | `backend/src/config/db.js:116-143` |
| **漏洞类型** | 逻辑缺陷（缓存永不失效） |
| **CVSS 评分** | 6.5 (Medium) |

**漏洞描述**：  
`invalidateTable` 方法按 `query:tableName` 前缀匹配缓存键，但 `generateCacheKey` 生成的缓存键是纯 MD5 哈希值（如 `a1b2c3d4...`），不包含表名前缀。导致 `invalidateTable` 永远无法精确匹配任何缓存键，只能走全量清除（fallback 逻辑）。但全量清除也因缓存键格式问题无法命中。

**影响范围**：
- 写操作（INSERT/UPDATE/DELETE）后缓存未失效
- 用户看到的是过期数据
- 数据一致性问题贯穿整个系统

**修复方案**：
1. 缓存键格式改为 `query:tableName:sha256hash`（可读前缀+哈希）
2. 新增 `extractTableName` 函数从 SQL 中提取表名
3. `invalidateTable` 按 `query:tableName:` 前缀精确匹配

**修复状态**：✅ 已修复

### 1.9 JWT_SECRET 空值校验缺失 + 密码强度不足

| 字段 | 详情 |
|------|------|
| **漏洞编号** | SEC-009 |
| **风险等级** | 🔴 严重 |
| **文件位置** | `backend/src/controllers/authController.js:24/31` + `L437` |
| **漏洞类型** | 认证绕过 + 弱密码策略 |
| **CVSS 评分** | 9.8 (Critical) |

**漏洞描述**：  
1. `generateAccessToken` 和 `generateRefreshToken` 直接使用 `process.env.JWT_SECRET` 签名，若环境变量未设置（值为 `undefined`），所有 Token 使用 `undefined` 作为密钥签名，攻击者可轻易伪造任意用户 Token
2. 注册接口密码验证仅要求 `isLength({ min: 6 })`，但 `security.js` 中已实现更强的 `validatePasswordStrength`（8位+大小写+数字），两者不一致导致安全策略形同虚设

**影响范围**：
- JWT_SECRET 未设置时，攻击者可伪造任意用户身份（包括管理员）
- 弱密码策略允许用户设置 `123456` 等简单密码，易被暴力破解

**修复方案**：
1. 模块加载时调用 `validateJwtSecret()` 校验 JWT_SECRET 非空且长度 >= 32 位
2. 注册验证中密码规则改为 `isLength({ min: 8 })` + `custom(validatePasswordStrength)`
3. 用户名验证增加 `matches(/^[a-zA-Z0-9_]{3,20}$/)` 格式限制

**修复状态**：✅ 已修复

## 五、P1 中优先级修复（已全部完成）

### 5.1 authMiddleware.js L1 缓存使用 lru-cache 限制内存上限

| 字段 | 详情 |
|------|------|
| **漏洞编号** | SEC-010 |
| **风险等级** | 🟡 中 |
| **文件位置** | `backend/src/middleware/authMiddleware.js:31` |
| **漏洞类型** | 资源泄漏（内存无限增长） |

**漏洞描述**：  
L1 内存缓存使用原生 `new Map()` 存储用户信息，无容量上限。高并发场景下，活跃用户信息持续积累，内存占用无限增长，可能导致 OOM。

**修复方案**：  
安装 `lru-cache` 包，将 `Map` 替换为 `LRUCache`，设置 `max: 5000` 条目上限和 `ttl: 60s` 自动过期。

**修复代码**：
```javascript
const { LRUCache } = require('lru-cache');
const l1Cache = new LRUCache({
  max: 5000,
  ttl: 60 * 1000,
  updateAgeOnGet: true,
});
```

**修复状态**：✅ 已修复

### 5.2 rateLimiter.js Redis 客户端优雅关闭

| 字段 | 详情 |
|------|------|
| **漏洞编号** | SEC-011 |
| **风险等级** | 🟡 中 |
| **文件位置** | `backend/src/middleware/rateLimiter.js` |
| **漏洞类型** | 资源泄漏（连接未释放） |

**漏洞描述**：  
限流器模块没有提供 Redis 客户端关闭方法，进程退出时 Redis 连接未被优雅释放，可能导致 Redis 服务端残留大量 TIME_WAIT 连接。

**修复方案**：  
新增 `closeRateLimiter()` 函数，在进程退出时调用 `redisClient.quit()` 优雅关闭，失败时回退到 `redisClient.disconnect()` 强制断开。
