# 性能基准测试指南

## 基准测试目标

### API响应时间

| API端点 | 目标P95 | 目标P99 |
|---------|---------|---------|
| `GET /api/farm/lands` | <200ms | <500ms |
| `POST /api/crops/plant` | <300ms | <800ms |
| `POST /api/crops/harvest` | <300ms | <800ms |
| `GET /api/user/data` | <150ms | <400ms |
| `POST /api/auth/login` | <300ms | <600ms |

### 并发能力

| 指标 | 目标 |
|------|------|
| 并发用户数 | 100+ |
| 每秒请求数(RPS) | 500+ |
| 错误率 | <1% |

## Artillery测试配置示例

### 1. 基础测试配置

```yaml
config:
  target: "http://localhost:3000"
  phases:
    - duration: 60
      arrivalRate: 10
      name: "Warm up"
    - duration: 120
      arrivalRate: 50
      name: "Sustained load"
  defaults:
    headers:
      Content-Type: "application/json"

scenarios:
  - name: "Farm page load"
    flow:
      - get:
          url: "/api/farm/lands"
  - name: "Crop plant"
    flow:
      - post:
          url: "/api/crops/plant"
          json:
            cropId: 1
            landNum: 1
```

### 2. 运行测试

```bash
# 保存为farm-test.yml
artillery run farm-test.yml
```

### 3. 生成报告

```bash
artillery run farm-test.yml --output results.json
artillery report results.json
```

## 关键性能指标

### 1. 响应时间
- P50：中位数
- P95：95%请求在该时间内
- P99：99%请求在该时间内

### 2. 吞吐量
- 请求总数
- 每秒请求数(RPS)

### 3. 错误率
- 成功率
- 各类错误分布

### 4. 资源使用
- CPU使用率
- 内存使用率
- 数据库连接数

## 性能优化建议

### 如果响应时间慢
1. 添加数据库索引
2. 使用Redis缓存
3. 优化SQL查询
4. 考虑异步处理

### 如果并发不够
1. 检查数据库连接池
2. 优化Redis连接
3. 使用负载均衡
4. 考虑水平扩展

### 如果错误率高
1. 检查超时设置
2. 添加重试机制
3. 优化资源释放
4. 检查错误日志

## 与CI/CD集成

在GitHub Actions中添加性能测试（可选）：

```yaml
performance-test:
  runs-on: ubuntu-latest
  needs: [backend-test]
  if: github.ref == 'refs/heads/main'
  
  steps:
    - uses: actions/checkout@v4
    - name: Setup Node.js
      uses: actions/setup-node@v4
      with:
        node-version: '20'
    - name: Install Artillery
      run: npm install -g artillery
    - name: Run performance test
      run: artillery run backend/tests/performance/farm-test.yml
```

## 性能测试最佳实践

### 1. 测试环境一致性
- 与生产环境配置接近
- 固定硬件规格
- 相同数据库大小

### 2. 基准线建立
- 在代码变更前运行一次
- 记录基准结果
- 变更后对比

### 3. 多次测试
- 运行多次取平均值
- 避免单次测试偏差

### 4. 隔离测试
- 不与其他测试同时运行
- 避免资源竞争

## 数据库性能优化

### 1. 检查慢查询

```sql
SELECT query, mean_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;
```

### 2. 添加索引

```sql
-- 为常用查询添加索引
CREATE INDEX IF NOT EXISTS idx_player_lands_player_id ON player_land_status(player_id);
```
